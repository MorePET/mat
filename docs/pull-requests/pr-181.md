# [PR 181](https://github.com/MorePET/mat/pull/181) feat: 3.9.0 — Aluminum 6061 alias + typo-tolerant search

Closes #178, #179. Two UX fixes surfaced via live MCP testing of \`pymat-mcp 0.1.1\` against Claude Code.

## #178 — \`pymat[\"Aluminum 6061\"]\` now resolves

Pure data fix: \`[aluminum.a6061]\` parent was misnamed \`\"Aluminum 6061-T6\"\` (the temper-baked form). Renamed parent to \`\"Aluminum 6061\"\`; T6 child keeps the composite name. Mirrors the long-shipped stainless pattern (parent = alloy designation, children = composite-named variants). 7075 was already correct — 6061 was the outlier.

## #179 — typo-tolerant search

Scorer rewrite: replace \`if token in target\` (substring containment) with \`rapidfuzz.fuzz.partial_ratio >= 75\` per token. The \"fuzzy\" search was misnamed — it did substring matching only.

\`\`\`
3.8: pymat.search(\"Stinless Stl 304\") → []
3.9: pymat.search(\"Stinless Stl 304\") → [s304, s303]
\`\`\`

Conjunctive contract preserved (every token clears threshold somewhere); weighted by similarity for ranking. Backwards-compatible — substring containment always scores 100 under partial_ratio, so the new path strictly widens hits without dropping existing matches.

## New runtime dep: \`rapidfuzz>=3.0\`

Pure-Rust wheel, broad platform support, no compile burden. ~300 KB.

## Tests

- \`+4\` in \`TestAlloyParentNaming\`: parent/temper hierarchy, distinct nodes, 7075 pattern consistency
- \`+4\` in \`TestTypoTolerance\`: one-char typo finds 304, transposed-digit finds 6061, conjunctive rejection holds with one fuzzy + one unrelated token, single-char drop resolves
- \`test_human_word_matches_name\` widened: \`aluminum\` legitimately surfaces \`alumina\` (related ceramic) under fuzzy — pin only that the aluminum tree appears
