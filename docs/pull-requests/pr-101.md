# [PR 101](https://github.com/MorePET/mat/pull/101) feat: mat-vis-client 0.6.3 substrate compat (3.5.0)

## Summary

- Bump `mat-vis-client>=0.5.0` → `>=0.6.3`. Substrate moved to per-file HF commits ([mat-vis 0.6.0 / ADR-0007 + ADR-0012](https://github.com/MorePET/mat-vis/releases/tag/v0.6.0)) and the index entry schema reshuffled.
- Rewrite `pymat.vis.search` as a thin forwarder to `mat_vis_client.search`. Upstream now does category/scalar/tier filtering + scoring; we keep only the `tags` post-filter (not in upstream).
- ~50 lines of duplicated filter/score logic deleted.

## Why

mat-vis-client 0.6.3 indexes nest scalars under `entry["mat_vis"]["pbr"]` and category under `entry["mat_vis"]["category"]`. Our previous `vis.search` read top-level keys, so under the new schema it would silently return `[]` for any category/scalar query.

- [x] `pytest -q` → 387 passed, 18 skipped
- [x] live texture fetch (`stainless.s304.vis.textures`) round-trips against `v2026.04.2`

## Notes

- New `tier` kwarg on `vis.search` (default `"1k"`) reflects upstream gating semantics.
- Test mocking switched from full `MatVisClient` mocks to monkeypatching `vis._client_search` — cleaner boundary; upstream owns the filter logic and tests it.
- `get_manifest()` schema check updated for v3 (`sources` field, not `tiers`).
