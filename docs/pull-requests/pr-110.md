# [PR 110](https://github.com/MorePET/mat/pull/110) feat: 3.7.0 — Material.copy/with_vis + Vis.override bug fixes

## Summary

Closes the registry-singleton mutation hazard end-to-end. 3.6.0 shipped \`Vis.override\` for the visual side; 3.7.0 fixes two bugs in that method, adds \`Material.copy()\` / \`Material.with_vis(vis)\` for the parent layer, and updates docs/example to teach the safe pattern as canonical.

All issues identified by independent implementation / DX / coverage reviews of 3.6.0:

|---|---|---|

| #104 | \`Vis.override(finishes=...)\` stored caller dict by reference | **Fixed** |
| #105 | Missing test coverage (adapter round-trip, tier preservation, finishes= map, None reset, etc.) | **Fixed** — 16 new tests (43 total) |

## Notable design decisions

- **\`with_vis\` deep-copies the supplied vis** — defends against \`m.with_vis(m.vis)\` aliasing the registry singleton. Same class of bug \`Vis.override\` had at the level above; same defense.
- **\`copy()\` zeros the texture cache explicitly** — \`deepcopy\` bypasses \`Vis.__post_init__\` (it goes through \`__reduce_ex__\`), so the cache must be cleared after copy.
- **\`copy()\` preserves \`parent\` by reference** — for the property-inheritance fallback chain. Mutating \`c.parent.<anything>\` still affects the registry; documented as a footgun, not yet protected.
