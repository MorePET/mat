# [PR 114](https://github.com/MorePET/mat/pull/114) feat: pymat-mcp v0.1.0 — standalone MCP server

Closes #113. Standalone PyPI package exposing the curated material registry to AI agents that talk MCP but don't run Python in the user's env.

End-user UX:

\`\`\`json
{ "mcpServers": { "pymat": { "command": "uvx", "args": ["pymat-mcp"] } } }
\`\`\`

That's the entire install. \`uvx\` fetches \`pymat-mcp\` (which transitively pulls \`py-materials\`) and runs in an isolated venv per session.

## What's in v0.1

10 read-only tools — \`search_materials\`, \`get_material\`, \`list_categories\`, \`list_grades\`, \`list_finishes\`, \`compute_mass\`, \`get_appearance\`, \`to_threejs\`, \`to_gltf\`, \`compare_materials\`. All catch+translate errors into \`{"error": "...", "did_you_mean": [{key, name}, ...]}\` envelopes; never raise.

## Pre-tag review fixes

Three independent reviews caught real issues before tagging:

| Source | Fix |
|---|---|
| DX | \`did_you_mean\` returns \`[{key, name}]\` not just names → agent retries with canonical key |
| DX | Param naming consistency (\`material\` everywhere; was a mix of \`key_or_name\`/\`material\`/\`keys\`) |
| DX | \`get_material(domains=, include_vis=)\` for compact payloads — halves typical token count |
| DX | \`list_finishes\` added (was forcing agents to call \`get_appearance\` and parse) |
| Scope | \`py-materials>=3.7.0\` floor (was 3.8.0; we don't use VisDeltas) |
| Scope | \`mcp<2\` upper cap |
| Scope | LICENSE in sdist |
| Scope | \`--version\` CLI flag for config-debug round-trip |
| Scope | Concurrency:\`pypi-publish\` on both publish workflows |
| **Scope** | **\`ci.yml\` runs MCP tests on PRs** — closed the "tests-only-on-tag" gap that would have shipped breakage straight to PyPI |
